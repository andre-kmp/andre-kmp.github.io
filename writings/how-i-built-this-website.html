How I built this website

July 2026.

The previous version of this website was very beautiful, but relied on tons of AI generated code (no shame) to come to life. Once it was finished, I couldn't publish any of what I had planned because, even though I was proud of it, I felt like that none of it belonged to me.

Feelings are keys for needs. Many months later I began, without any expectations, working through the getting started modules on MDN. What a surprise when I grasped, semantically, how to write and read HTML and CSS code as a whole!

I came back to my website's directory, deleted the HTML and CSS files and started over from scratch. The process can be broken down into four interchangeable stages:

  1. Conception
  2. File structuring (HTML, CSS, JS)
  3. Content creation
  4. Responsiveness

Conception

I started by listing how I wanted my website to look and sketched out some ideas for the homepage and content pages in a notebook. To explore color palettes, I visited Color Hunt. I began coding to see the ideas on the screen. This was when I started selecting typography at Google Fonts.

Here are requirements I listed right at the start:

  • A website to promote and document my projects and ideas
  • Needs to be durable and perpetual
  • I also want to introduce myself on it
  • I want it to be navigable on both desktop and mobile

File structuring (HTML, CSS, JS)

Understanding the concepts below made all the difference in coding fluently:

  1. Margin, padding and border
  2. Selector types (classes, elements, ids), pseudo-classes, and pseudo-elements
  3. Emphasis, importance, and associated semantics

There is no need to include a description of the code, as it is available in the repository. I avoided JavaScript as much as possible, given its steeper learning curve (FTOWBIM is the only exception).

Content production

This step is the core of why I call the process interchangeable. I decided to write, asap, two text pages, so I could be able to test the behavior of the Writings page. As I reread the texts, I refined them slightly until considered them finished. Since this is a 100% personal project, I feel free to work at my own pace.

I decided to write in English, even though it is not my native language. I always write the initial draft on my own, without the aid of any tool. For corrections, I recur to Google Translator (another interchangeable process), and less so, to the Merriam-Webster dictionary. I am commited to typing everything out, avoiding copy and paste, and it seems my English writing skills have improved during the course of July 2026.

Responsivity

This was the last stage I worked on, and it was easier than I expected. My main technical reference was a responsive web-design guide published by Designveloper. Responsive web-design is a complex topic, and small decisions can alter the entire user experience. The main dilemmas I faced here were:

  1. How to handle overflow? (increase the box height or add a scrollbar?)
  2. If the window width decreases, what should be reduced: the margin width, or the content width?
  3. Which font-sizes are fixed? And which can scale up or down, and to what extent?

I use Mozilla Firefox as my browser. The Responsive Mode (Ctrl + Shift + M) was very useful for simulating the mobile experience.

I am very proud of the results. The main takeaway is that a well structured CSS saves a lot of time down the line. In the future, I want to include a language switcher to pt_BR, my native language, and a light-mode option.

I am neither a designer or a front-end developer, but I find web-design delightful. If you'd like to ask, point out a mistake or simply chat about this topic, feel free to e-mail me at andreklaic@gmail.com.
